    Memory AI Lab architecture

    Scopo: definire il progetto finale abbastanza chiaramente da guidare le lezioni, senza implementarlo in anticipo.

    Obiettivo del sistema

    Memory AI Lab riceve memorie testuali, le valida, le arricchisce, le indicizza e permette retrieval e valutazione offline. Il sistema deve funzionare localmente. Il cloud e' un'estensione documentata, non una dipendenza base.

    Componenti logici

    ingestion
-> validation
-> cleaning
-> classification
-> importance scoring
-> embedding
-> graph extraction
-> retrieval
-> evaluation
-> monitoring reports

    Contratti minimi

    Ingestion

    Input: CSV o JSONL locale con memory_id, text, timestamp.

    Output: record normalizzati e report data quality.

    Validation

    Controlla:

    • campi obbligatori;
    • timestamp parsabile;
    • id univoci;
    • testo non vuoto;
    • range importance se presente.

    Classification

    Classi minime:

    • episodic
    • semantic
    • preference
    • unknown

    Baseline richiesta prima di LLM: modello semplice o regole testabili.

    Importance scoring

    Baseline locale:

    • feature testuali semplici;
    • recency;
    • euristiche documentate.

    Nessun punteggio deve essere presentato come verita' assoluta.

    Embedding and retrieval

    Prima baseline:

    • vettori piccoli o embedding model leggero;
    • cosine similarity;
    • metriche Recall@K e MRR.

    LLM o Gemma arrivano solo dopo metriche baseline.

    Graph memory

    Nodi minimi:

    • memory;
    • entity;
    • event;
    • time.

    Relazioni minime:

    • mentions;
    • occurred_at;
    • related_to.

    Il grafo deve supportare almeno una query utile, non solo visualizzazione.

    Structured extraction

    Prima regole o modello piccolo; poi modello open. Ogni estrazione generativa deve avere metriche:

    • JSON valid rate;
    • field accuracy;
    • hallucinated field rate.

    Evaluation

    Report minimi:

    • classification metrics;
    • retrieval metrics;
    • structured extraction metrics;
    • data quality summary;
    • drift simulation quando introdotto monitoring.

    Directory attese

    src/memory_ai/
  data_cleaning.py
  schema.py
  validation.py
  classification.py
  importance.py
  embeddings.py
  graph.py
  retrieval.py
  evaluation.py
  monitoring.py

    Questa e' una direzione architetturale, non un permesso a creare tutti i file in anticipo. Ogni file nasce quando una lezione ne ha bisogno.

    Decisioni ancora aperte

    • Libreria per schema runtime: dataclass, Pydantic o JSON Schema.
    • Embedding baseline locale.
    • Preset KerasHub/Gemma compatibile.
    • Formato artifact del grafo.
    • Soglie minime per PASS finale.

    Ogni decisione deve essere risolta nella lezione che la introduce e registrata nel relativo review report.
